# CLAUDE.md — Правила для работы с проектом GTO_App

## Архитектура приложения

Приложение состоит из **двух HTML-страниц**, работающих с **общей базой данных IndexedDB**:

- `GTO_App/index.html` — Дашборд (школьный реестр, управление сессиями)
- `GTO_App/workspace.html` — Рабочее пространство (выбор участников, проверка, выгрузка Excel)

## КРИТИЧЕСКОЕ ПРАВИЛО: Синхронизация index.html и workspace.html

**При ЛЮБОМ изменении в `index.html` ОБЯЗАТЕЛЬНО проверить `workspace.html` и наоборот:**

1. Если добавляется новый `<script>` в index.html — проверить, нужен ли он в workspace.html
2. Если добавляется новый `<link>` стилей — проверить оба файла
3. Если меняется модель данных в `school-storage.js` — проверить, что `app.js` (workspace) и `school-roster.js` (dashboard) оба обновлены
4. Если добавляются новые вкладки/категории на дашборде — добавить соответствующие вкладки в выбор участников workspace
5. Если меняется IndexedDB схема (версия, stores) — оба HTML используют одну БД, обе страницы должны работать

### Общие скрипты между страницами:
- `scripts/core/session-storage.js` — обе страницы
- `scripts/core/school-storage.js` — обе страницы
- `assets/vendor/xlsx.full.min.js` — обе страницы
- `assets/vendor/exceljs.min.js` — обе страницы

## Формат данных

- **Документ** в экспорте: серия и номер через пробел (например "II-ЕР 567761"), БЕЗ символа №
- **Адрес**: без "Тольятти" (город фильтруется), формат: "ул Название, д. X, кв. Y"
- **Дата рождения**: хранится ISO (YYYY-MM-DD), отображается ДД.ММ.ГГГГ
- **Пол**: "Мужской" / "Женский"
- **УИН**: формат XX-XX-XXXXXXX

## Источник данных

Школьный реестр (IndexedDB) — **единственный источник данных** для рабочего пространства.
Workspace автоматически загружает данные из реестра при открытии.
Шаг "Подготовка данных" (загрузка файлов) пропускается если реестр заполнен.
